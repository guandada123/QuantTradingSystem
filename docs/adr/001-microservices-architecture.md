# ADR-001: 微服务架构选型

**状态**: ✅ 已采纳  
**日期**: 2026-06-05  
**决策者**: QuantTradingSystem 团队

---

## 背景

A股超短线量化交易系统需要同时处理三类截然不同的工作负载：

1. **策略研究**: 下载历史K线 → 计算技术指标 → 回测 → 生成信号（计算密集型，分钟级批处理）
2. **交易执行**: 接收信号 → 风控检查 → MiniQMT下单 → 持仓管理（IO密集型，秒级实时，需要与券商客户端长连接）
3. **AI调度**: 调用LLM API → 多智能体分析 → 飞书告警（网络IO密集型，异步并发）

单体架构下，三类负载争抢同一进程的 CPU、内存、GIL。策略回测会阻塞交易执行，飞书告警延迟受LLM API响应影响。

---

## 决策

**采用微服务架构，拆分为 3 个独立服务 + 基础设施。**

```
┌───────────────┐    ┌──────────────┐    ┌──────────────┐
│ strategy-     │    │ execution-   │    │ ai-scheduler │
│ service       │───▶│ service      │◀───│              │
│ FastAPI       │    │ FastAPI      │    │ FastAPI      │
│ Celery Worker │    │ MiniQMT长连接 │    │ LLM Client   │
└───────┬───────┘    └──────┬───────┘    └──────┬───────┘
        │                   │                    │
        └───────────────────┼────────────────────┘
                            │
            ┌───────────────┼───────────────┐
            │               │               │
    ┌───────▼──┐   ┌───────▼──┐   ┌───────▼──┐
    │PostgreSQL│   │  Redis   │   │RabbitMQ  │
    └──────────┘   └──────────┘   └──────────┘
```

**服务通信**:
- 同步: REST API (FastAPI + httpx)
- 异步: RabbitMQ 消息队列 (策略 → 执行 → AI通知)
- 缓存: Redis (信号缓存、会话状态)

**服务职责边界**:

| 服务 | 职责 | 技术特性 |
|------|------|----------|
| strategy-service | 数据采集、指标计算、回测、信号生成 | Celery Worker 异步执行回测，不阻塞 API |
| execution-service | MiniQMT 连接、订单管理、风控、持仓 | 独立的 MiniQMT 长连接，避免与其他服务争夺GIL |
| ai-scheduler | LLM调用、多智能体编排、飞书告警 | 独立扩缩容，可按需增加LLM并发 |

---

## 后果

### ✅ 正面

1. **独立扩缩容**: 交易时段加大 execution-service 机器，收盘后加大 strategy-service 计算资源
2. **技术栈灵活性**: 各服务可使用最适合的语言/库（目前均为Python，但未来策略引擎可用Rust重写核心算法）
3. **故障隔离**: execution-service 崩溃不会中断 strategy-service 的回测任务
4. **团队并行**: 三个开发者可独立工作于不同服务

### ❌ 负面

1. **运维复杂度**: 3个服务 = 3个Dockerfile + 3个K8s Deployment + 监控 ×3
2. **网络延迟**: 服务间 HTTP 调用引入 1-5ms 延迟（对秒级交易可接受）
3. **数据一致性**: 分布式事务需要最终一致性设计（当前通过RabbitMQ消息确认保证）
4. **调试复杂度**: 需要分布式追踪（TraceIDMiddleware 已部分解决）

### ⚖️ 权衡

相对于单体架构，微服务在开发前三周增加了约 30% 的样板代码量（Dockerfiles、K8s YAML、服务发现）。但对于团队协作和长期维护，这个投入值得。

当前 3 服务拆分是**最小可行拆分**。未来可能的进一步拆分：
- 风控服务（如果风控逻辑复杂度显著增长）
- 数据服务（如果需跨策略共享缓存层）
